# 원론적 해결 원칙 (전역)

> 현상 → 근본 원인 → 구조적 수정. 증상 패치는 다음 날 같은 이슈를 만든다.

1. **현상 먼저 파악**: 왜 이 이슈가 발생했는가? 구조적 원인은?
1b. **끝까지 추적**: 사용자 클릭 → URL → 프론트 state → API 호출 → 응답 → UI 렌더링 전체 경로를 코드 레벨로 따라간 뒤에만 "근본 원인"이라 판단할 것.
1c. **수정 후 실행 검증 필수**: "코드 논리상 맞으니까 될 것이다"는 검증이 아님. 실제 실행으로 확인.
2. **재발 방지 룰화**: 수정 후 반드시 룰 추가
3. **단일 소스 원칙**: 동일한 값은 단일 config에서 읽는다
4. **예외 처리 명시**: bare `pass`/`except: ...` = 오류 마스킹
5. **새 기능 = 새 룰**: 깨질 수 있는 경우를 미리 룰로 명문화
6. **기능 회귀 방지**: 변경 전 기존 기능 나열 → 약화 식별 → 사용자 확인
7. **가동 중 서버 재시작 금지**: 외부 API startup dependency 위험. 로컬 테스트 완료 + 트래픽 없는 시간에만
8. **새 코드 배포 전 로컬 검증**: 빌드 통과만으로 부족. API 직접 호출, 런타임 검증 필수
9. **인프라 환경 인지**: 이 머신 = Mac Mini (IP 172.30.1.16). 자기 자신에게 SSH 시도 금지
